# 2026-03-07

## Windows selection-context UX decision

- Manual Windows QA of the `1.4.16` editor hardening work showed the core problem was broader than caret math alone.
- Full-text editor overlays that repaint blurred editor content on top of a native `textarea` remained unstable on Windows.
- Observed failure modes during manual testing:
  - caret/text drift in longer wrapped documents
  - duplicated or smeared text when inline prompt overlays remained enabled
  - duplicated or misaligned text when the persistent selection overlay remained enabled
- The macOS path did not show the same level of instability during prior manual testing, so the product decision is platform-specific for now.

## Approved product direction

- Product requirement retained:
  - users must be able to select text, move focus to the prompt input, and still clearly understand what text the LLM will edit
- Windows implementation direction:
  - keep native editor text/caret rendering stable
  - stop relying on blurred full-text editor overlays to preserve selection context after blur
  - preserve selection intent in app state (`tab`, `selectionStart`, `selectionEnd`, selected text, and validity against current content)
  - show selected-text context beside the prompt rather than by repainting the selected text inside the editor
- macOS implementation direction:
  - keep the current persistent in-editor selection highlight behavior unless later QA shows comparable instability there

## Why this direction was chosen

- A plain `textarea` cannot style substrings directly, which is why Ghost Writer originally used painted text overlays.
- On Windows, those overlays do not stay reliably aligned with the native textarea text metrics during blur/selection flows.
- The real UX requirement is not "a blurred selection must stay highlighted inside the editor."
- The real UX requirement is "the user must know exactly what text the prompt will edit after moving to the prompt input."
- A prompt-adjacent selected-text preview satisfies that requirement with less platform risk on Windows.

## Follow-up implementation target

- Windows-only selection system target:
  - native editor text remains visible and authoritative
  - native selection remains while the editor is focused
  - once focus moves to the prompt input, Windows shows a compact selected-text preview near the prompt
  - prompt submission rewrites the saved selected range if it is still valid
  - if the selection becomes invalid because content changed, the app should fail safely and ask the user to reselect
